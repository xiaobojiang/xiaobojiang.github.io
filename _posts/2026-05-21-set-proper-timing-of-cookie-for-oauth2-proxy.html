🧩 Issue

We’re using oauth2-proxy with our IDM to authenticate users and get JWT access tokens and cookies.

However, we observe the following behavior:

  • Users can log in successfully.

  • Their JWT access token expires after 1 hour (as set by the IdM).

  • The session cookie remains valid much longer, so oauth2-proxy still considers the session authenticated.

  • When calling backend APIs, the forwarded access token is expired → API requests fail with 401 even though the user appears logged in.

🔍 Root Cause

This happens because of how oauth2-proxy manages session cookies vs tokens:

  1. Session cookie lifetime is independent from JWT expiryBy default, oauth2-proxy issues a session cookie that can be valid far longer than the access token. The proxy only checks the cookie — not whether the JWT is still valid — before letting the user through.

  2. jwt access token expires sooner than the cookieThe backend receives the expired JWT and rejects it, even though oauth2-proxy still sees the session as valid.

  3. Token refresh only happens under certain conditionsoauth2-proxy will refresh tokens only if:

    • A refresh token is present,

    • The cookie_refresh threshold is reached,

    • And a new request arrives after that threshold.

  4. Without a refresh token or proper settings, the access token won’t get refreshed automatically.

  5. Changing settings can invalidate existing sessionsWhen cookie_refresh/cookie_expire are shortened, or the cookie_secret changes, previously issued cookies may no longer match new rules and force users to log in again.

✅ How to Fix It

To ensure users don’t keep a valid session while sending expired JWTs to your backend:

1. Enable and use the refresh token

Make sure your IdM issues a refresh token (e.g., include offline_access scope if needed).

Without a refresh token, oauth2-proxy cannot renew the access token.

2. Tune cookie_refresh relative to access token lifetime

Because your access token is valid for 1 hour, set:

cookie_refresh = "55m"

This tells oauth2-proxy to attempt refresh just before expiration so the JWT remains valid.

3. Set cookie_expire to match refresh token TTL

Your IdM refresh token expires in 48 hours, so:

cookie_expire = "48h"

This allows the session cookie to remain valid as long as the refresh token does, letting oauth2-proxy continue refreshing.

4. Understand refresh behavior

  • oauth2-proxy only refreshes when a request arrives aftercookie_refreshis reached.

  • It does not refresh in the background without a request.

  • If the refresh token itself expires, users will have to re-login.

5. Config updates and session invalidation

If you shorten cookie values or rotate the cookie secret:

  • Existing cookies may be considered invalid on the next request.

  • That forces users to authenticate again.
