Let's start from below scenario, imagine your company has two git servers, one is in intranet, address is:

http://192.168.1.101

and the other is in internet, address is:

https://git.iotcolon.com

Suppose we ae working on a project, called "startgit". I am the owner, name is xiaobo.

Now I am gonna create a git repository for this project.

After setting up framework and creating folder "startgit", I will do git initialization and commit

$ cd startgit
$ git init
$ git add .
$ git commit -m "first commit"

Now in order to collaborate with team members, we need to push it to server, the first step is to add remote address.

First, we need to create a repository in the intranet git server, and add this to your local git repository, we call it intragit.

$ git remote add intragit http://192.168.1.101/xiaobo/startgit.git

Then, for the public git server, we can also add it for later push, we create it in the public git server, and add it with name publicgit.

$ git remote add publicgit https://git.iotcolon.com/xiaobo/startgit.git

Now, we have already added two remote servers, we can do push.

we can push it first to intranet server's master branch, (-u intragit, means push to intranet git server, here intragit is the alias for the long url: http://192.168.1.101/xiaobo/startgit.git).

$ git push -u intragit master

Later, when the work is done and proper test has been passed, we can push the stable branch to public server (-u publicgit)

$ git push -u publicgit stable

Note: the formula is git push -u [remote server alias] [branch name]

Here, we may found this command is different from what we normally observe in git turtorial.

$ git push -u origin master

well, now you must get the point, "intragit", "publicgit" and "origin" are just the same thing, they represent the remote git urls with a human-friendly name.

In fact, upper commands that push to remote server, can also be called like:

$ git push http://192.168.1.101/xiaobo/startgit.git master
$ git push https://git.iotcolon.com/xiaobo/startgit.git stable

Using the alias can really improve efficiency, because human cannot remember the full url. the concept is similar to relationship between Domain name and IP.

Furthermore, this alias are also reflected in our folder structure.

If we pull code from intragit it actually updates data to .git/refs/remote/intragit folder from intranet git server.

In .refs/remote, we can also found intragit and publicgit folder, so corresponding action on data happens here.

We can also check these in the /git/config

$ cat .git/config                                                                                                                         
[core]
        repositoryformatversion = 0
        filemode = true
        bare = false
        logallrefupdates = true
        ignorecase = true
        precomposeunicode = true
[remote "intragit"]
        url = http://192.168.1.101/xiaobo/startgit.git
        fetch = +refs/heads/*:refs/remotes/intragit/*
[remote "publicgit"]
        url = https://git.iotcolon.com/xiaobo/startgit.git
        fetch = +refs/heads/*:refs/remotes/publicgit/*

take intragit as example, parameter url is git repository address. Parameter fetch in this example is:

+refs/heads/*:refs/remotes/intragit/*

What does this line means? The part at left side of colon "refs/heads" is local working folder, and part at right side("refs/remotes/intragit") is the remote repo ('s local cache folder).

If we do fetch from public remote git server http://192.168.1.101/xiaobo/startgit.git, it will pull code update to corresponding branch in folder refs/remotes/intragit. when we do git merge, it will merge code at corresponding branch in refs/remotes/intragit/ to corresponding branch in refs/heads.

Come back to the first question in the title, what is origin in git? When you clone, git needs to create a folder in .git/refs/remotes/, and cache data. origin is just the default name, meaning where is comes from.
