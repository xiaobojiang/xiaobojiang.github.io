Let’s try something cool in this post, we will use one Raspberry Pi connected to AWS iot core via MQTT. The AWS provides easy to use and scalable platform for devices to connect, the core service is AWS IOT. With device connected, we can also use data analytics and other value added service to devices.
MQTT is a industry standard, and it message model is broker-client model, all the client subscribe and publish topics in the broker. MQTT is light weight and highly stable, which makes it very suitable for IOT applications.

Figure 1, example of MQTT messaging, source: https://mqtt.org

AWS IoT provides MQTT based connectivity:

Figure 2, supported protocol and ports, source: docs.aws.amazon.com


In this post, we will use MQTT over port 8883, whose authentication is based on X509 certificates. Let’s try one publish and subscribe example.

Before we do the coding, we need to make some configuration the AWS IoT: 

  1. Create the policy, as the starting point, we can make the policy simple as:

Figure 3. Create an example policy

{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "iot:Connect",
        "iot:Receive",
        "iot:Publish",
        "iot:Subscribe"
      ],
      "Resource": "*"
    }
  ]
}

Here, connect/receive are the action that MQTT requires, check here for more details: https://docs.aws.amazon.com/iot/latest/developerguide/iot-policy-actions.html, and full action resources are listed here:https://docs.aws.amazon.com/iot/latest/developerguide/iot-action-resources.html With the combination, the policy can specify what actions can do on what resources.

In the upper policy, basically it is not a secure one, it enables device can connect/receive/publish/subscribe to any topics/resources. 

  1. Next is to create a thing object in the IOT console

Figure 4. Create a thing

Specify a name for this thing instance, please be noted that this name cannot be modified after creation. Keep additional configuation and Device Shadow with default values.

Figure 5. properties of a new thing

And in the device certificate window, choose auto-generate a new certificate.

Figure 6. Create certificate for the created thing, choose recommended

And remember to download the device cert (rename to device.pem.crt), puiblic key (rename to public.pem.key), private key (rename to private.pem.key) and RSA CA cert (Amazon-root-CA-1.pem). These certs and keys will be used in client. If you check the certs assigned, you will find the root CA expires on  2038-1-17

Figure 7. sample ca certificate from AWS IOT

  The device cert expires on 2049-12-31

Figure 8. sample device certificate from AWS IOT

Root CA actually has shorter valid time than device certificate, which means you need to refresh certs in your devices after 2038… that’s already long time after.

  1. all the resource have been created in the AWS cloud.  Next optional step is to improve the policy with policy variables

AWS IoT Core defines policy variables that can be used in AWS IoT Core policies in the Resource or Condition block. When a policy is evaluated, the policy variables are replaced by actual values.

For example, if a device is connected to the AWS IoT Core message broker with a client ID of 100-234-3456, the iot:ClientId policy variable is replaced in the policy document by 100-234-3456

Below is a better policy, which resticts devices with client id prefix “gateway-“ can connect to broker, and all the device can only pub/sub topic with rules brightsite/gateway/${iot:ClientId}/*. This makes the device can do action at topic within his client id as prefix, blocks it listening message from other client.

As a note, only one device with same client id can access AWS iot.
Policy:

{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "iot:Publish",
        "iot:Receive"
      ],
      "Resource": [
        "arn:aws:iot:eu-central-1:accountid:topic/brightsite/gateway/${iot:ClientId}/*"
      ]
    },
    {
      "Effect": "Allow",
      "Action": [
        "iot:Subscribe"
      ],
      "Resource": [
        "arn:aws:iot:eu-central-1:accountid:topicfilter/brightsite/gateway/${iot:ClientId}/*"
      ]
    },
    {
      "Effect": "Allow",
      "Action": [
        "iot:Connect"
      ],
      "Resource": [
        "arn:aws:iot:eu-central-1:accountid:client/gateway-*"
      ]
    }
  ]
}

Next part is on device side

I choose Raspberry PI 4B as the client. there are few steps:

1, Install the latest Raspbian, it contains python 3.9.2 as default, execute “apt update” and “apt upgrade” to update system.

2, Install necessary packages: 

$ sudo apt-get install git
$ sudo apt-get install cmake 
$ sudo apt-get install libssl-dev
$ pip install awsiotsdk


3, pull the sample code from github

$ cd ~ 
$ git clone https://github.com/aws/aws-iot-device-sdk-python-v2.git


4, create a folder called certs, and put previous keys and certs inside

5, change the code, and update the policy and topics to updated policy.

Example pubsub.py code, I made few small changes on it client id and pub/sub topic, so it can works together with the improved policy:

#only difference is pasted
if __name__ == '__main__':
    # Spin up resources
    event_loop_group = io.EventLoopGroup(1)
    host_resolver = io.DefaultHostResolver(event_loop_group)
    client_bootstrap = io.ClientBootstrap(event_loop_group, host_resolver)

    #override args parameter
    args.client_id = 'gateway-'+ str(uuid4())
    args.topic = 'brightsite/gateway/'+args.client_id+'/ping'

6, execute the code  

$ python3 pubsub.py --root-ca ~/certs/Amazon-root-CA-1.pem --cert ~/certs/device.pem.crt --key ~/certs/private.pem.key --endpoint your-iot-endpoint —-port 8883

It use the certificates in the cert folder, and further more,And you could see outputs like:

Figure 9. console output of pubsub.py code

This means that the device connected the AWS broker, and it can publish and receive the message on the same topic.
As a final note, during the code execution, if it complains client connect closed, most of the time it is due to policy issue. AWS iot core rejects the connection, and pub/sub, check the policy and adjust.
