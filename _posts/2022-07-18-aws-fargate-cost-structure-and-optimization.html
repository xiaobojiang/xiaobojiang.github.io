Explore the fargate cost structure.

AWS Fargate pricing is calculated based on the vCPU and memory resources used from the time you start to download your container image until the Amazon EKS Pod terminates, rounded up to the nearest second.

It is claimed that the cost is calculated based on the actual cpu and memory usage[1].

Alright, we can use kubectl top command to verify the utilization done by each resource(pod, daemonsets) on the node.

Eg:

Take the first pod as example, now it consumes 0.002 core cpu and 305Mi.

Now the question:

In every hour, how much you need to pay for the first pod?

My early thought is, rounded up the resource, so CPU is 0.25 core, and memory is 512Mi. Then, the cost is 0.04656*0.25 + 0.00511*0.5 = 0.014195 per hour.

Well, if you think that is the case, then you will get very surprised at the monthly bill.

What’s the actual story?

First, in kubernetes, user will specify the resources, in requests and limits.

Eg:

        resources:
            requests:
              memory: "512Mi"
              cpu: "250m"
            limits:
              memory: "1Gi"
              cpu: "250m"

Second, the AWS fargate cost is based on the fargate node that assigned to this pod.

The fargate node has certain support configurations [1]:

CPU                Memory Values

0.25 vCPU      0.5 GB, 1 GB, and 2 GB

0.5 vCPU        Min. 1 GB and Max. 4 GB, in 1 GB increments

1 vCPU           Min. 2 GB and Max. 8 GB, in 1 GB increments

2 vCPU           Min. 4 GB and Max. 16 GB, in 1 GB increments

4 vCPU           Min. 8 GB and Max. 30 GB, in 1 GB increments

You can check out the pod actual configuration in the AWS console.

According to my experiment, the initial assigned configuration is based on the resource requested. And memory wise, it always assigns a level up than you request.

Eg, it assign 0.25Core+1G ram, which is 512Mi more than I requested, even the actual consumption is 325Mi. I guess ram configuration is what I requested plus some overhead (logging), so rouned up to next level.

And further more, if you request 2 core 2G ram, and you will get at least 2core 4G ram fargate node. Although you really consumes little on the node, the cost will be so high. So it works like virtual managed EC2, with certain type to choose from. (similar to t3 small/medium)

Once you know how it works, there are few tipcs to save cost for our EKS fargate cluster:

1, do not claim luxury resource which beyonds our actual consumption

2, use tools, eg kubectl top to monitor the resources. Note, monitoring service is not default available on AWS EKS, we can install it easily.

3, we can also use Vertical pod autoscaler or Horizontal pod autoscaler to scale up resource when required [2,3].
