Recent days, I am playing APIs with my Hue bridge and lights.

I've setup a jupyter notebook container runtime to write notebook and debug and explore the APIs.

Then I found that my code cannot mdns discover hue bridge, while after a while, trying the same on the host it works..

I recalled that my notebook is running in docker network, which is a vlan from the host, while the mdnd request cannot pass through.

Did few research with AI, find two solutions:

  • using --network=host

  • creating a macvlan

Let me explain more on macvlan

🔍 Understanding macvlan Networking

  • Direct Network Access: Containers on a macvlan network have their own IP addresses and can communicate directly with other devices on the same network segment.

  • No Port Publishing: The standard -p or --publish flags used in Docker to map container ports to host ports are ineffective with macvlan networks. This is because containers are already accessible via their own IP addresses, making port publishing unnecessary.

  • Host-Container Communication: By default, the host cannot directly communicate with containers on a macvlan network due to network isolation.

🔄 Accessing macvlan Containers from the Host

To enable the host to communicate with containers on a macvlan network, you can create a macvlan interface on the host itself. Here’s how:

  1. Create a macvlan Network:

docker network create -d macvlan \

  --subnet=192.168.1.0/24 \

  --gateway=192.168.1.1 \

  -o parent=<network_interface_name> macvlan_net

  1. Create a macvlan Interface on the Host:

ip link add macvlan_host link <network_interface_name> type macvlan mode bridge

ip addr add 192.168.1.100/24 dev macvlan_host

ip link set macvlan_host up

This assigns the IP 192.168.1.100 to the host’s macvlan interface, allowing it to communicate with containers on the macvlan_net network.

  1. Run a Container on the macvlan Network:

docker run -it -d --name jupyter  --network=macvlan_net -p 8888:8888 quay.io/jupyter/minimal-notebook

  1. The container is now accessible at a separate ip in lan, such as 192.168.1.100, and the host can communicate with it via the macvlan_host interface.

Now, it works, my bridge is found with mdns discovery!

🔐 Security Considerations

  • Isolation: macvlan networks provide isolation between the host and containers, which can enhance security but may complicate communication.

  • Access Control: Ensure that appropriate firewall rules and access controls are in place to prevent unauthorized access to containers on the macvlan network.

  • Monitoring: Regularly monitor network traffic and container activity to detect any unusual behavior.

By configuring a macvlan interface on the host, you can facilitate communication between the host and containers on a macvlan network, enabling functionalities like mDNS discovery while maintaining network isolation.
