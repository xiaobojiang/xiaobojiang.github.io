When you have a EKS cluster, and running services, there is always requirement that EKS pods needs to access other aws resources, eg, S3.

One common way, is you create a IAM user with programtic access, and add policy to specify the resource go/no-go, and later use the access id and key in the AWS SDK to call APIs.

I would not say this is bad, but this brings a lot of chanlendges: 1, management of those crendentials, 2, you may need to create many IAM users to ensure control granularity, 3, it is not easy to automate.

Actually, I found that there is a more elegant way in aws to do this: service account.

People who are familiar with Kubernetes shall also know this idea[1], A service account provides an identity for processes that run in a Pod.

You can associate an IAM role with a Kubernetes service account. This service account can then provide AWS permissions to the containers in any pod that uses that service account. With this feature, you no longer need to provide extended permissions to the Amazon EKS node IAM role so that pods on that node can call AWS APIs [2]. From my test, pod with service account, it is calling web-identity to sts service get API access credentials.

In comparison, you can also configure IAM role on nodes (EC2) [3], so that all pods on the node have the access. I would not recommend this, personally it is not so scalable and not good granularity.

The IAM roles for service accounts feature provides the following benefits:

Least privilege — By using the IAM roles for service accounts feature, you no longer need to provide extended permissions to the node IAM role so that pods on that node can call AWS APIs. You can scope IAM permissions to a service account, and only pods that use that service account have access to those permissions.

Credential isolation — A container can only retrieve credentials for the IAM role that is associated with the service account to which it belongs. A container never has access to credentials that are intended for another container that belongs to another pod.

Auditability — Access and event logging is available through CloudTrail to help ensure retrospective auditing.

Try it out with S3

After go through the concept, let's take one example to illustrate the how-to: S3 access.

Here, also share the process mainly with CDK python.

Step 1, create a S3 bucket.

block all public access and make encrpytion as KMS_MANAGED.

        appS3 = s3.Bucket(self,
            "AppDataBucket",
            block_public_access=s3.BlockPublicAccess.BLOCK_ALL,
            encryption=s3.BucketEncryption.KMS_MANAGED
        )

Step 2, create corresponding access policy. Here I referred to example S3 policy of aws [4]

allow all access to this bucket.

Note: later I found I cannot list buckets, is it due to only one bucket is listed in allowed resource.

        

        appS3_policy_document_json = {
            "Version": "2012-10-17",
            "Statement": [{
                "Effect": "Allow",
                "Action": [
                    "s3:*"
                ],
                "Resource": [
                    "arn:aws:s3:::"+appS3.bucket_name,
                    "arn:aws:s3:::"+appS3.bucket_name+'/*'
                ]
            }]
        }
        appS3_policy_document = iam.PolicyDocument.from_json(appS3_policy_document_json)

        appS3_policy = iam.Policy(self,
            'appS3Policy',
            policy_name='appS3Policy',
            document=appS3_policy_document
        )

Step 3, for the required namespaces, create service acccount, and associate policy to service account's role.

#use service account way, , you need to specify sa in pod yaml, eg: serviceAccountName: serviceaccount-appdev
        for namespace in namespaces:
            #create service account
            eksS3ServiceAccount = ekscluster.add_service_account(
                'serviceaccount-'+namespace,
                name='serviceaccount-'+namespace,
                namespace=namespace
            )
            eksS3ServiceAccount.role.attach_inline_policy(bseS3_policy)

Step 4: create a python pod in the namespace, and try to access with boto3.

boto3 is the aws python sdk, refer to [5].

in below deployment, please note the line of serviceAccount, and fill the name of created service account.

#pod deployment yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: eks-iam-test-s3
  namespace: appdev
spec:
  replicas: 1
  selector: 
    matchLabels:
      app: eks-iam-test-s3
  template:
    metadata:
      labels:
        app: eks-iam-test-s3
    spec:
      serviceAccountName: serviceaccount-appdev
      containers:
      - name: eks-iam-test
        image: python:3.9
        command: ["/bin/sh"]
        args: ["-c", "while true; do sleep 10;done"]
      restartPolicy: Always


# deploy exec into your python pod
$ kubectl apply -f s3-pod.yaml
$ kubectl exec -it eks-iam-test-s3 -n appdev bash
$ pip install boto3
$ python

Try following code in python pod, you can see the files in the S3 bucket, generate presigned url and download the files to your pod.

If your program works the same, congrats, your pod now have access to S3 bucket, and enjoy!

#example python commands in inteactive shell
$ import boto3
$ client = boto3.client('s3')
$ response=client.list_objects(Bucket='your bucket name')
$ print(response)
$ client.generate_presigned_url('get_object',{'Bucket': 'your bucket name', 'Key': 'yourfilename'},HttpMethod="GET")
$ client.download_file('your bucket name', 'your file name', 'saved file name')
