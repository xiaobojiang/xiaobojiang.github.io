This is the second post on CDK, previous post talks about CDK basics and installation.

In this post, we will use CDK to create and provision a EKS fargate cluster.

Here, I will assume you already have your CDK environment ready and running.

To make it simple, I will put all the code in one stack, in practice, you can split different functions into different stacks or python files.

Let's take a look at the architecture we want to achieve. All resources are inside a VPC, and for the sake of security, all EKS cluster's resources are in the private subnets of VPC.

Image Source: https://aws.amazon.com/quickstart/architecture/amazon-eks/

VPC

Create a VPC in CDK is rather simple, specify the name, cidr range, number of zones.

sample code:

         vpc = ec2.Vpc(self,
            'my-vpc', # VPC id
            cidr=cidr_range, # CIDR
            max_azs=3,  # 3 aza
            nat_gateways=1 # create NAT Gateway
        )

EKS fargate cluster

Then, create a master role for eks cluster.  In order to issue kubectl commands against an EKS cluster, the mastersRole must be explicitly defined.

        eks_master_role = iam.Role(self, 'eksMasterRole',
            role_name='eksAdminRole',
            assumed_by=iam.AccountRootPrincipal()
        )

Now, you can create a eks cluster, specify that we create a fargate cluster, and subnets type is private, also make the endpoint public accessible:

         ekscluster = eks.FargateCluster(self, 
            'myCluster',  # cluster id
            vpc=vpc,  # ref VPC
            cluster_name='myCluster',
            version=eks.KubernetesVersion.V1_19,  # K8S version
            masters_role=eks_master_role, # master's IAM Role
            endpoint_access=eks.EndpointAccess.PUBLIC, # no need nodes
            vpc_subnets=[{'subnet_type':ec2.SubnetType.PRIVATE}]
        )

Here, AWS provides two kinds of eks cluster, one is common ec2 based, second is fargate one.

AWS Fargate is a technology that you can use with Amazon ECS to run containers without having to manage servers or clusters of Amazon EC2 instances. With Fargate, you no longer have to provision, configure, or scale clusters of virtual machines to run containers. This removes the need to choose server types, decide when to scale your clusters, or optimize cluster packing.

few differences:

1. Performance - The difference here is due to the fact that with an EC2, the compute is already accounted for so all the service needs to do is directly assign the pods to one of the nodes registered and actively reporting in your cluster. Fargate allocates the right amount of compute, eliminating the need to choose instances and scale cluster capacity and each task or pod in its own kernel providing the tasks and pods their own isolated compute environment, however, a larger provisioning time would be added to your workloads as the provisioning happen 'on-the-fly'. This means that you will need to architect your application to be able to handle these delays.

2. Pricing model/Cost - In relation to this topic, please note that according to our documentation, the amount of $0.10 is charged per hour for each Amazon EKS cluster that you create. Besides that, When using EC2 to run your Kubernetes workers nodes allows you to optimize price by taking advantage of billing models such as spot instances, or reserved instances. In this case, amounts are charged for the instances and for the resources associated with them, such as EC2 instances or EBS volumes. On the other hand, using Fargate, pricing is calculated based on the vCPU and memory resources used from the time you start to download your container image until the Amazon EKS pod terminates, rounded up to the nearest second.

3. fargate also has its limitations: cluster cannot use GPU, daemonset is not supported. fargate uses sidecar similar technology to enable logging

Here, I want to try out the new tech, I choose to use fargate cluster. however, the configuration is similar, using fargate, you do not need to add a NodeGroup as EC2 cluster.

Fargate Profile

For fargate cluster, you need to create a fargate profile, and add namespaces into it, so the pods defined in these namespaces can be assigned to fargate nodes.

when the cluster was created, it contains a default fargate profile, and namespaces is "default" and "kube-system".

Once the fargate profile is created, you cannot add namespaces to it.

If you face this case, create a new profile.

        namespaces = ['dev','qa']
        #add profile for user namespaces
        selectors = []
        for namespace in namespaces:
            selectors.append(eks.Selector(namespace=namespace))
            #create namespace in kubectl
            nsManifest = {
                "apiVersion":"v1",
                "kind": "Namespace",
                "metadata": {"name": namespace}
            }
            ekscluster.add_manifest("namespace-"+namespace, nsManifest)

        myfargateprofile=ekscluster.add_fargate_profile(
            'appfargateprofile',
            fargate_profile_name='appFargateProfile',
            selectors=selectors
        )

Metrics Service

By default, aws cluster does not contain metrics server.

you can add with following lines:

        #deploy metrics service
        metrics_manifest_url = 'https://github.com/kubernetes-sigs/metrics-server/releases/latest/download/components.yaml'
        metric_manifest = list(yaml.safe_load_all(requests.get(metrics_manifest_url).text))
        ekscluster.add_manifest('metrics-server', *metric_manifest)

AWS Load Balancer Controller

If you want to expose your service, an ingress is required, here AWS provides three ways: application load balancer, network load balancer and classic load balancer. difference can be found: https://aws.amazon.com/elasticloadbalancing/features/

By installing Aws Load Balancer Controoller, you can config at ingress which type you would like to use.

Here, let's install Aws Load Balancer Controller, few steps in below code:

1, Create an IAM role and annotate the Kubernetes service account that's named aws-load-balancer-controller in the kube-system namespace

2, create an IAM policy for the AWS Load Balancer Controller that allows it to make calls to AWS APIs on your behalf. 

3, attach the policy created in step 2 to the service account aws-load-balancer-controller

4, using helm chart to create Aws Load Balancer Controller

Logging to CloudWatch

Steps referred to https://docs.aws.amazon.com/eks/latest/userguide/fargate-logging.html

here, also shell scritps are heavily used, not all code in CDK:

1, Create a dedicated Kubernetes namespace named aws-observability.

#file of aws-observability-namespace.yaml


kind: Namespace
apiVersion: v1
metadata:
  name: aws-observability
  labels:
    aws-observability: enabled

2, Create a ConfigMap with a Fluent Conf data value to ship container logs to a destination. Fluent Conf is Fluent Bit, which is a fast and lightweight log processor configuration language that is used to route container logs to a log destination of your choice.

# file: aws-logging-cloudwatch-configmap.yaml


kind: ConfigMap
apiVersion: v1
metadata:
  name: aws-logging
  namespace: aws-observability
  labels:
data:
  output.conf: |
    [OUTPUT]
        Name cloudwatch_logs
        Match   *
        region ${REGION}
        log_group_name fluent-bit-cloudwatch
        log_stream_prefix from-fluent-bit-
        auto_create_group true

3, use shell script to apply them in eks cluster


export REGION=eu-central-1

info_line "enable logging for fargate pods, to cloudwatch"

kubectl apply -f ${current_dir}/k8s/fargate-logging/aws-observability-namespace.yaml

envsubst '${REGION}' < ${current_dir}/k8s/fargate-logging/aws-logging-cloudwatch-configmap.yaml  | kubectl apply -f -

4, Create a CloudWatch IAM policy, and Attach the IAM policy to the pod execution role specified for your Fargate profile. 

        #enable cloudwatch logs
        cloudwatch_policy_document_json ={
            "Version": "2012-10-17",
            "Statement": [{
                "Effect": "Allow",
                "Action": [
                    "logs:CreateLogStream",
                    "logs:CreateLogGroup",
                    "logs:DescribeLogStreams",
                    "logs:PutLogEvents"
                ],
                "Resource": "*"
            }]
        }

        cloudwatch_policy_document = iam.PolicyDocument.from_json( cloudwatch_policy_document_json )

        cloudwatch_policy = iam.Policy(self,
            'CloudWatchLogEKSPolicy',
            policy_name='CloudWatchLogEKSPolicy',
            document=cloudwatch_policy_document
        )

        myfargateprofile.pod_execution_role.attach_inline_policy(cloudwatch_policy)
