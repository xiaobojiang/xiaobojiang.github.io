Recently, I has one requirement that one app container shall be abled to run on cloud server (which is x64 arch /amd64), and also on box like raspberry pi (arm64 arch).

This is called multiple platform or multi-arch support.

Following is the technology history roadmap from developer.ibm.com:

"Projects that do support multiple platforms have resorted to using image names to indicate which platform an image was built for. For example, my-cool-app-ppc64le:latest; or creating a Docker hub namespace for each architecture, such as ppc64le/my-cool-app.

To alleviate these issues, the Docker community has come up with what is currently called a manifest list, also nicknamed a multi-arch image, or fat manifest. An individual manifest describes the contents of one image, and a manifest list enables you to group multiple images together. After the creation of a manifest list, anyone (from the project owners to their user-base) can use the manifest list name where they once used an individual image name. When a user runs a docker pull or a docker run command, the Docker engine does the work of selecting which image to pull based on the operating system and architecture on which it is running. As a result, a project's users no longer have to worry about finding the name of the image that will work for their platform. The project can have a single name, my-cool-app:latest"

So the popular way is to using manifest to group images together, and docker engine will based on its architecture to pull right image.

We do not need to build the tool from scrath, docker has a tool called buildx can help us.

First, make sure following preequisites are met:

1, docker version must be larger than 19.03

2, turn on docker's experiment feature.

# edit ~/.docker/config.json, add one line
experimental": "enabled

Then we need to install buildx on the build machine

  • Download the lastest version and right arch bin file from https://github.com/docker/buildx/releases 

#rename the file
$ mv buildx-v0.8.2.linux-amd64 docker-buildx

#grant execution right to the file
$ chmod +x docker-buildx

#on my machine, this cli-plugins folder is not present, create the folder
$ mkdir -p ~/.docker/cli-plugins 

#move the file
mv docker-buildx ~/.docker/cli-plugins

  • Verify that buildx is running properly

$ docker buildx ls

  • Create a driver that support different arch

$ docker buildx create --driver docker-container --platform linux/amd64,linux/arm64 --name remote-builder
#and use this builder
Docker buildx use remote-builder

  • Check again

The builder shall with star symbol, and platforms already includes the ones we intend to support.

Finally, try to build your first image that support multi-arch with buildx

Single line command, this command can help us build images, prepare manifests and push to registry:

$ docker buildx build -o type=registry -f "Dockerfile path" --build-arg version=${version} --platform linux/arm64/v8,linux/amd64 --tag "registry.cn-shanghai.aliyuncs.com/mycoolapp:${tag}" .

Here, we can use --push as short hand of -o type=registry

--platform specifies the platform we want to support, add more if you need.

The dockerfile is the same, however, better to check whether your base image in it support your desired paltforms.

buildx has many more option that you can explore, find them here.

Check the manifest of builded images.

So we can pull the image of different arch with single tag.

That's pretty cool!
