When configuring the proxy_pass directive in Nginx, the presence or absence of a trailing slash in the URL affects how paths are handled. If a trailing slash is added after the URL, it indicates an absolute root path. If there is no trailing slash, it indicates a relative path, including the matched part of the location block.

Let's explore four different configurations exmaples using the URL http://192.168.1.1/proxy/test.html.

Example 1:

location /proxy/ {
    proxy_pass http://127.0.0.1/;
}

Result: Proxied URL: http://127.0.0.1/test.html

Note: here proxy_pass has no trailing slash, make the location absolution, /proxy/ is replaced with absolute address http://127.0.0.1/ + test.html

Example 2 (without the trailing slash):

location /proxy/ {
    proxy_pass http://127.0.0.1;
}

Proxied URL: http://127.0.0.1/proxy/test.html

Note: no trailing slash at proxy_pass, just appending the location to the end of target server address http://127.0.0.1 + /proxy/test.html

Example 3:

location /proxy/ {
    proxy_pass http://127.0.0.1/aaa/;
}

Proxied URL: http://127.0.0.1/aaa/test.html

Note: trailing slash make the absolute path, there /aaa replace the /proxy. Final url is http://127.0.0.1/aaa/test.html

Example 4 (without the trailing slash):

location /proxy/ {
    proxy_pass http://127.0.0.1/aaa;
}

Proxied URL: http://127.0.0.1/aaatest.html

Note:

When a client requests http://192.168.1.1/proxy/test.html, Nginx matches this request to the location /proxy/ block.

URL Handling by Nginx:

  1. Matching Location Block:

    • The request URI /proxy/test.html matches the location /proxy/ block.

  2. Proxy Pass Directive without Trailing Slash:

    • proxy_pass http://127.0.0.1/aaa;

    • Since there's no trailing slash after /aaa, Nginx will append the remaining part of the client request URI to the URL specified in proxy_pass.

    • The remaining part of the request URI, after /proxy, is /test.html.

  3. Constructed Proxied URL:

    • Nginx constructs the proxied URL by concatenating http://127.0.0.1/aaa with /test.html.

    • This results in http://127.0.0.1/aaatest.html.

The fourth example is really above my head. So I setup an example server for testing upper case.
with nginx config:

 server {

        listen 80;
        location /proxy/ {
          proxy_pass http://127.0.0.1:8080/aaa;
          }
}

Then write a python server to print out the request url:

import http.server
import socketserver

PORT = 8080

class RequestHandler(http.server.SimpleHTTPRequestHandler):
    def do_GET(self):
        print(f"Requested URL: {self.path}")
        super().do_GET()

with socketserver.TCPServer(("", PORT), RequestHandler) as httpd:
    print(f"Serving on port {PORT}")
    httpd.serve_forever()

Finally, test the url

$ curl localhost:80/proxy/test.html

and we get this log from python test server. Wow, yes, it did append the test.html after path `aaa`!
