 Secret is a blob of data, such as a password, SSH private key, SSL certificate, or another piece of data that should not be transmitted over a network or stored unencrypted in a Dockerfile or in your application’s source code. You can use Docker secrets to centrally manage this data and securely transmit it to only those containers that need access to it.

Ways to create secret in docker:

$ docker secret create my_secret ./secret.json

or docker compose way

---
services:
  frontend:
    image: awesome/webapp
    secrets:
      - server-certificate
secrets:
  server-certificate:
file: ./server.cert
# external: true

Example usage in swarm mode:

In swarm mode, it protect secret better, inspect does not reveal more information

docker secret create my_secret ./client.crt

docker stack deploy -c docker-compose.yml MyApplication

currently, docker only support create standalone secret in swarm mode[1].

It is also mentioned that docker-compose now also supports secret.

Let’s check it:

“secrets grants access to sensitive data defined by secrets on a per-service basis. Two different syntax variants are supported: the short syntax and the long syntax.

Compose implementations MUST report an error if the secret doesn’t exist on the platform or isn’t defined in the secrets section of this Compose file. ”[2]

Docker-compose secret (not in swarm mode) need to specify local file, which means it is still in some easy-find place. Like below example.

---
services:
  frontend:
    image: awesome/webapp
    secrets:
      - server-certificate
secrets:
  server-certificate:
    file: ./server.cert

Conclusion: Docker secret can protect senstive information, and does not need to hardcode it. Currently, you can use it in docker swarm mode.
