OpenAI’s vision docs describe how images are tiled into 512×512 patches, with a base 85 tokens plus 170 tokens per tile in detail:"high"; detail:"low" is a flat 85 tokens regardless of size. Most calculators above encode that logic. (Always check the model you’re using—some variants may diverge.)

What do “patches” / “tiles” mean here?

  • Think of a large image (e.g. 1024×768 px, or larger) as being composed of smaller square blocks (“tiles” / “patches”) of size 512×512 px.

  • The idea is: instead of processing the entire large image as one monolithic input, the model (or the billing logic) splits it into chunks of size up to 512×512, because that’s a convenient granularity for processing or costing.

  • If your image fits entirely within one 512×512 block (i.e. width ≤ 512 and height ≤ 512), then n = 1. If it’s larger, you need multiple tiles to “cover” it.

  • Mathematically, for an image of width W and height H, you compute: $$\text{tiles\_width} = \lceil W / 512 \rceil,\quad \text{tiles\_height} = \lceil H / 512 \rceil$$ Then the total number of tiles = tiles_width × tiles_height.

  • Example: for a 600 × 400 image:

    • tiles_width = ceil(600 / 512) = 2

    • tiles_height = ceil(400 / 512) = 1

    • total tiles = 2 × 1 = 2


  • So under “high detail mode” that image would cost ≈ 85 + 170 × 2 = 425 tokens by that formula.

This “tiling” approach is common in vision systems to keep computations / memory manageable and structured, and also for billing — it’s easier to say “each 512×512 patch costs 170 tokens” than to do a fine per-pixel or per-feature cost.

What is “detail = high / detail = low” meaning in this context?

In the OpenAI vision / image billing logic (as inferred from documentation + community observations):

  • detail = “low” means a low resolution / low fidelity mode. In that mode, the system uses a coarser / simpler representation of the image. In billing terms, the image is treated as though it has only the base cost of 85 tokens, regardless of how big the image is (i.e. even if the image is much larger than 512×512). This is a way to save token cost (and usually compute) when you don’t need high detail.

  • detail = “high” means a high resolution / high fidelity mode. In that mode, the model (or billing logic) does break the image into 512×512 tiles and charges 170 tokens per tile, plus the base 85 tokens. So the total is: $$\text{Total tokens} = 85 + 170 \times \text{(number of 512×512 tiles)}$$ This lets a high-detail image cost more (because you’re effectively representing it at a finer granularity).

  • The “auto” setting (if supported) might let the system pick low vs high based on the image size or expected usage.

Why a base 85 tokens? And why 170 per tile?

  • The base 85 tokens is like a “fixed surcharge” for having an image at all — perhaps to represent a coarse / thumbnail / overview embedding of the image. Even in “high” mode, the model needs an overall summary or low-res version, so that cost is baked in.

  • The 170 tokens per 512×512 tile represents the “extra detail cost” for each chunk. If there are more tiles (i.e. a larger image needing more chunks), you pay more.

  • The numbers (85, 170) come from OpenAI’s documentation / pricing examples, and from what community users reverse-engineered based on token usage reports. For instance, the official OpenAI vision docs show an example: “4 tiles are needed … so final token cost is 170 × 4 + 85” for some high-detail image. 

  • Also, in community discussions: “Each tile provides 170 tokens … total tokens = 85 + 170 * n” is a repeatedly cited formula. 

Extra complications and caveats (from user reports)

  • In many community reports, token usage doesn’t always scale exactly as the formula — for instance, images beyond certain sizes are downscaled first, or the “short side is clamped to 768 px” before tiling. 

  • Some users report that even with 512×512 images, when “detail = auto” or when no detail parameter is given, the system still charges the higher “high detail / tiled” cost instead of just the 85 token low cost. 

  • There may be metadata, padding, or internal rounding that causes small deviations from the ideal formula. 

Example formula:

def calculate_image_tokens(width, height, detail="high"):
    if detail == "low":
        return 85  # minimal cost in some modes

    # Clamp / scale down if bigger than 2048 × 2048
    max_size = 2048
    if width > max_size or height > max_size:
        aspect = width / height
        if aspect > 1:
            width = max_size
            height = int(max_size / aspect)
        else:
            height = max_size
            width = int(max_size * aspect)

    # If width & height both exceed some “minimum” threshold, scale the short side to 768
    min_size = 768
    aspect = width / height
    if width > min_size and height > min_size:
        if aspect > 1:
            # width is longer side
            height = min_size
            width = int(min_size * aspect)
        else:
            width = min_size
            height = int(min_size / aspect)

    tiles_w = math.ceil(width / 512)
    tiles_h = math.ceil(height / 512)
    return 85 + 170 * (tiles_w * tiles_h)
