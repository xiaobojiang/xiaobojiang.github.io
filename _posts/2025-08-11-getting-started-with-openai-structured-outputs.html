How to enforce predictable, schema-based LLM responses with Pydantic and.parse()

📌 Introduction

In August 2024, OpenAI launched Structured Outputs, a feature that lets you lock LLM responses to a strict format you define.

Why is this a big deal?

Because LLMs love being “creative” — sometimes too much. Even when you ask for a list or a single word, they might give you a sentence, bullet points, or something else entirely.

With Structured Outputs, you define the schema once, and every response must match it. No more broken JSON. No more regex hacks.

🛠 1. Setting Up

You’ll need:

  • Python 3.10+

  • An OpenAI API key

  • openai and pydantic installed

pip install -U openai pydantic

Test your setup:

from openai import OpenAI

client = OpenAI()

response = client.chat.completions.create(

    model="gpt-4o-mini",

    messages=[

        {"role": "system", "content": "You are a helpful assistant."},

        {"role": "user", "content": "Say hello!"}

    ],

    max_tokens=5

)

print(response.choices[0].message.content.strip())

→ "Hello!"

📐 2. Define Your Schema with Pydantic

Structured Outputs require a Pydantic model to describe the exact response structure.

Example: a simple sentiment schema ⬇️

from pydantic import BaseModel

from typing import Literal

class SentimentResponse(BaseModel):

    sentiment: Literal["positive", "negative", "neutral"]

This enforces:

  • Only the sentiment field

  • Only these 3 allowed values

📦 3. Using .parse() to Enforce the Schema

Instead of create(), use .parse() and pass response_format=YourModel.

response = client.beta.chat.completions.parse(

    model="gpt-4o-mini",

    messages=[

        {"role": "system", "content": "Classify sentiment."},

        {"role": "user", "content": "The room was clean and the staff was friendly."}

    ],

    response_format=SentimentResponse

)

print(response.choices[0].message.parsed.sentiment)

→ "positive"

Now every result matches your schema. No extra text, no surprises.

🧩 4. Nested Schemas for Complex Data

Pydantic lets you define nested models for hierarchical data.

Example: Extract user info with addresses:

class Address(BaseModel):

    street: str

    city: str

    state: str

    zip_code: str

class UserInfo(BaseModel):

    name: str

    email: str

    phone: str

    addresses: list[Address]

LLM output will always match this structure — even for multiple addresses.

🔌 5. Function Calling + Structured Outputs

Before Structured Outputs, defining function call schemas meant hand-writing long JSON schemas.

Now:

class WeatherData(BaseModel):

    location: str

    unit: Literal["celsius", "fahrenheit"]

    condition: Literal["sunny", "cloudy", "rainy", "snowy"]

tools = [openai.pydantic_function_tool(WeatherData)]

This auto-generates a function-call-compatible schema for the API.

You just feed it to tools= and let the model produce structured arguments.

💡 6. Best Practices

✅ Keep schemas simple & specific

✅ Use Literal for enums

✅ Add field descriptions with Field()

✅ Handle refusals via .refusal attribute

✅ Use .parsed instead of manual JSON parsing

Example with descriptions:

from pydantic import Field

class Person(BaseModel):

    name: str = Field(..., description="Full name")

    age: int = Field(..., description="Age in years")

    occupation: str = Field(..., description="Current job title")

🎯 Conclusion

Structured Outputs make LLMs predictable, safe, and production-ready.

By pairing Pydantic models with .parse(), you can:

  • Guarantee output formats

  • Reduce prompt-engineering time

  • Simplify function calling
