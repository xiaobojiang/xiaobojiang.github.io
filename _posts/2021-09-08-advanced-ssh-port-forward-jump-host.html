SSH is a handy service that allows user connects to remote computer instance securely, and this is the common way that user logins to cloud EC2.
But ssh can do more, imagine following scenario I always encounter:

1, expose remote service to local computer

2, login to EC2 within private subnet / Intranet via a jump host


Port Forwarding


In my daily work, I have encountered those scenarios: 1, access Kubernetes dashboard without deploy ingress 2, access influx db management portal 3, access not public exposed debug APIs.

SSH can address those requirements.
SSH port forwarding is a mechanism in SSH for tunneling application ports from the client machine to server machine, or vice versa.

It can be used for adding encryption to legacy applications, going through firewalls, and some system admin use it to open backdoor into internal network from their home computer. 


Port forwarding has two types of usage: local forwarding and remote forwarding.


Local Forwarding


Local forwarding is used to forward a port from a client machine to server machine.

Basically, the ssh client listens for connections on a configured port, and when it receives a connection, it tunnels the connection to a ssh server. 
In figure 1, if user want to access web service on the ssh server, it can make the connections on local 8000 forwarded to remote server’s port 80.
In openssh, local portfowarding is configured using the -L option.
Following line can fulfill the purpose:

$ ssh -L 8000:127.0.0.1:80 user@ssh.server.address

Furthermore, local forwarding can also make local connections forwarded to service which remote ssh server can access. Eg, mysql database in private subnet.


Using this line: 

$ ssh -L 3306:intra.rds.address:3306 user@ssh.server.address

Figure 1: typical local forwarding scenario


Remote Forwarding


In Openssh, remote ssh port forwarding is specified by using -R option. For example:

$ ssh -R 8080:localhost:80 user@ssh.server.address

This allows anyone on the remote server connect to TCP port 8080 of remote server. The connection will be tunneled back to client host, and ssh client makes a TCP connection to port 80 on localhost. Any other host name or IP address could be used instead of localhost to specify the host to connect to.

Figure 2: remote port forwarding scenario


This can be helpful for giving someone on the outside access to an intranet web server. Be cautious, this could be done by an employee working at home, or by a malicious attacker.  


By default, OpenSSH only allows connecting to remote forwarded ports from the server host. However, the GatewayPorts option in the server configuration file sshd_config can be used to control this.

The following alternatives are possible:

GatewayPorts no

This prevents connecting to forwarded ports from outside the server computer.

GatewayPorts yes

This allows anyone to connect to the forwarded ports.

If the server is on the public Internet, anyone on the Internet can connect to the port.

GatewayPorts client

specifiedThis means that the client can specify an IP address from which connections to the port are allowed.

The syntax for this is:

$ ssh -R 52.191.20.31:8080:localhost:80 host147.aws.example.com

In this example, only connections from the IP address 52.194.1.73 to port 8080 are allowed.

Jump Host

Imagine the following scenario, if user want to login to server B which is inside private intranet, is it possible to do so?

Here jump host can help solve this issue, by adding a jump host in public subnet, user can login to B via the “jump” server A.

Figure 3: login to server B via jump host A

You can do this via two ways:

1, editing ssh config file, it is a file located at ~/.ssh/config.

Host serverB
      HostName 192.168.0.12
      User ec2-user
      IdentityFile /users/keys/serverB.pem
      ProxyJump serverA
Host serverA
      HostName jump.example.com
      User default 
      IdentityFile /users/keys/serverA.pem

Here, identity file is the private key for ssh identification.

Then, you can login to server B via:

$ ssh serverB

2, another is command line way

Openssh provides -J option, works like:

$ ssh B -J A

In most scenarios, you will use key-pair as identity for ssh.

Directly specifying private with ‘-i’ option fails, you will need to translate the -J into long-form proxy command.

$ ssh -o ProxyCommand="ssh -i /path/ServerA.pem -W %h:%p user@serverA_host" -i /path/serverB.pem user@serverB
