OpenClaw session management is simple once you separate session keys from session IDs.

A good working model is:

  • Create / use a session: run /session <name> in TUI. If that key does not already exist, OpenClaw starts using it as that session namespace. Session keys are stored as agent:<agentId>:<sessionKey>. So /session main expands to the current agent’s main session, while /session agent:other:main jumps across agents. 

  • List / switch sessions: use /sessions or press Ctrl+P to open the session picker. The picker shows sessions for the current agent only. 

  • Start TUI directly in a named session: openclaw tui --session bugfix. 

The part that often confuses people is /new vs /reset.

In the TUI docs, both are documented as resetting the session. In the session docs, exact /new or /reset starts a fresh session ID. That means they are effectively the same for clearing active context. The one important difference is that /new can also take a model argument, like /new claude or /new openai/gpt-5, to start the fresh session on a different model.

So the practical workflow is:

  • /session feature-x → create or move into a named working session

  • /sessions → see what exists

  • Ctrl+P → switch quickly

  • /reset → hard reset current conversation

  • /new → hard reset current conversation, optionally with a new model 

One more useful detail: OpenClaw persists sessions in two layers on the Gateway host:

  • sessions.json tracks metadata for each session key

  • <sessionId>.jsonl stores the actual transcript

That is why session management feels clean in the UI while still keeping durable history under the hood.

A compact version:

In OpenClaw, use /session <name> to create or enter a named session, /sessions or Ctrl+P to list and switch, and /reset or /new to start fresh. /new is basically /reset plus optional model switching. Session keys identify the workspace; session IDs identify the actual conversation instance.
